## 2. 사용자 유형 & 권한

### 사용자 구분

| 유형 | 로그인 방식 | 접근 화면 | 비고 |
| --- | --- | --- | --- |
| **자녀 (CHILD)** | 6자리 숫자 코드 입력 | 홈·퀘스트·챗봇·가챠·도감·스트릭 | 이메일·비밀번호 없음 |
| **부모 (PARENT)** | Google 로그인 (Firebase Auth) | 부모 대시보드·알림설정·자녀프로필관리 | 자녀 화면 접근 불가 |

### 계정 구조

```jsx
[부모 계정] ─── Google UID (Firebase)
     │
     └── [자녀 프로필] ─── 6자리 코드 (예: 482917)
              │
              ├── xp_data (totalXp·todayXp·weeklyXp·gachaPullCount)
              ├── pet_state (장착 펫 ID·pet.xp·단계)
              ├── streak (연속 학습 기록·보호권)
              ├── gacha_data (티켓·조각·보유 펫 목록)
              ├── mission_attempts / mission_daily_progress (미션 시도 및 일별 요약)
              ├── daily_quests (오늘 퀘스트 달성 현황)
              ├── weekly_quests (이번 주 퀘스트 달성 현황)
              └── achievements (달성한 업적 기록)
```

### 핵심 제약

- 만 14세 미만 → 부모가 Google 계정으로 먼저 가입 후 자녀 코드 발급 (개인정보보호법 제22조의2)
- 자녀 계정 저장 데이터: **닉네임 + 코드만** (실명·이메일·생년월일 수집 안 함)
- Supabase RLS(Row Level Security) 필수 적용 → 다른 자녀 데이터 접근 차단

```jsx
앱 첫 실행
  └─▶ 분기 선택 화면
        ├─ [부모로 시작] ─▶ Google Sign-In (Firebase Auth)
        │                      ├─ 성공 ─▶ 자녀 닉네임 입력
        │                      │          └─▶ POST /api/parent/register
        │                      │                └─▶ 6자리 코드 생성 & 저장
        │                      │                └─▶ 스타터 패키지 지급: tickets.normal += 3

        │                      └─ 실패 ─▶ 에러 토스트 + 재시도 버튼
        └─ [자녀 코드 입력] ─▶ 코드 입력 화면 (섹션 3-2)
```

```jsx
[신규 자녀 프로필 최초 생성 시 1회만 지급]

// → 가챠 3번 뽑기 가능 → 🥚 알 상태 펫 최소 1마리 획득
// → MVP에서는 뽑은 펫 중 첫 번째를 자동 장착
// → 중복 지급 방지: child_profiles.starterIssued = true 플래그로 체크
```

### 3-2. 자녀 로그인 플로우

```
6자리 코드 입력
  └─▶ POST /api/child/login { code: "482917" }
        ├─ 유효 ─▶ 자녀 세션 토큰 발급 → SharedPreferences 저장 → 홈 화면
        ├─ 무효 ─▶ "코드를 다시 확인해봐요" 토스트
        └─ 3회 연속 실패 ─▶ 30초 대기 타이머 표시 후 재활성화
```

### 3-3. 코드 재발급

```
부모 앱 > 자녀 프로필 > 코드 재발급 버튼
  └─▶ PUT /api/parent/child/{childId}/regenerate-code
        └─▶ 기존 코드 무효화 + 새 6자리 코드 생성
        └─▶ 기존 자녀 세션 강제 만료 (자녀가 새 코드로 재로그인 필요)
```

## 4. FEATURE 01 — 3단계 AI 학습 퀘스트 🎯

### 담당: BE (문한일) — API·DB·채점 로직 | FE (임다빈) — 화면·애니메이션

### 4-1. 개요

| 항목 | 내용 |
| --- | --- |
| 총 문제 유형 | 16개 (1단계 5개 + 2단계 6개 + 3단계 5개) |
| 문제 유형 | OX, 객관식, 빈칸채우기, 상황카드 (4종) |
| 문제 출처 | 고정 DB (MVP 목표 120문제) + GPT 동적 생성 하이브리드 |
| 단계 잠금 | 1단계: 없음 / 2단계: 1단계 3개↑ 완료 / 3단계: 2단계 4개↑ 완료 |

### 4-2. API 흐름

```jsx
[문제 요청]
GET /api/missions/{missionId}/questions
  Headers: Authorization: Bearer {childSessionToken}
  └─▶ Supabase: question_bank_safe 뷰에서 missionId 기준 문제 조회
  └─▶ 서버는 quizAttemptId 세션을 생성하고, 정확히 10문항만 반환
        ├─ 문제 충분 (≥10개) ─▶ 랜덤 10개 반환
        └─ 문제 부족 (<10개) ─▶ GPT API 호출로 동적 생성 후 반환
              └─ GPT 오류 시 ─▶ 고정 DB 문제로 폴백 (오류 로그 기록)

Response JSON:
{
  "missionId": "uuid",
  "quizAttemptId": "uuid",
  "questionCount": 10,
  "expiresAt": "2026-04-11T10:00:00Z",
  "questions": [
    {
      "id": "q_001",
      "type": "OX",          // OX | MULTIPLE | FILL | SITUATION
      "question": "AI는 항상 정확한 정보를 말한다",
      "options": null        // 객관식·빈칸·상황카드는 배열
      // ⚠️ answer, explanation 절대 미포함 — 클라이언트에 정답 노출 시 치트 가능
      // 정답/해설은 제출(POST) 후 results[] 배열에서만 반환
    }
  ]
}

[정답 제출]
POST /api/missions/{missionId}/submit
  Body: { quizAttemptId, answers: [{questionId, selected}] }
  └─▶ 서버에서 채점 (클라이언트 채점 X — 정답 노출 방지)
  └─▶ 10문항 기준 오답 3개 이상이면 미션 실패
  └─▶ 통과(score >= 8)일 때만 완료 처리 로직 실행 (→ 섹션 11 연쇄 이벤트 참고)

Response:
{
  "score": 8,
  "total": 10,
  "wrongCount": 2,
  "isPassed": true,
  "isPerfect": false,
  "equippedPetGrade": "EPIC",
  "bonusXp": 10,
  "bonusReason": "PET_RARITY_BONUS",
  "xpEarned": 20,
  "streakUpdated": true,
  "ticketEarned": 0,
  "levelUp": false,
  "results": [             // 제출 후에만 정답/해설 포함
    {
      "questionId": "q_001",
      "isCorrect": true,
      "explanation": "AI는 할루시네이션이 발생할 수 있어요"
    }
  ]
}
```

### 4-3. 문제 유형별 명세

| 유형 | 입력 | 화면 구성 | 채점 방식 |
| --- | --- | --- | --- |
| **OX** | O 또는 X 탭 | 질문 텍스트 + O/X 큰 버튼 2개 | answer(bool) 비교 |
| **객관식** | 보기 중 1개 탭 | 질문 + 보기 2~4개 버튼 | answer(string) 비교 |
| **빈칸채우기** | 단어 카드 탭 | 빈칸 문장 + 선택지 카드 | answer(string) 비교 |
| **상황카드** | 행동 카드 탭 | 시나리오 텍스트 + 카드 2~3개 | answer(string) 비교 + 해설 필수 |

### 4-4. 보상 정책

```jsx
세션 1개 제출 시:
  totalQuestions = 10
  wrongCount = totalQuestions - score
  isPerfect = (wrongCount == 0)   // 퍼펙트 표시는 여전히 0오답만 허용

  if (wrongCount >= 3) {
    // 미션 실패
    xp += 0
    streak.todayCount 변화 없음
    missionComplete = false
    // 결과 화면에서 오답 해설만 제공, 재도전 유도
  } else {
    // 미션 통과
    xp += 10

    // 장착 펫 희귀도 보너스 XP (퍼펙트 판정과 별도)
    switch (equippedPet.grade) {
      case NORMAL:
        if (wrongCount == 0) xp += 10
        break
      case RARE:
        if (wrongCount <= 1) xp += 10
        break
      case EPIC:
        if (wrongCount <= 2) xp += 10
        break
      case LEGEND:
        if (wrongCount <= 2) xp += 15
        break
    }

    streak.todayCount += 1
    if (streak.todayCount == 1)   // 오늘 첫 미션 완료
      streak.continuousDays += 1
      → 연쇄 이벤트 트리거 (섹션 11 참고)
  }

  // ⚠️ 데일리 퀘스트 3개 완료 티켓은 섹션 11 연쇄 이벤트 ⑧에서 통합 처리
  // 여기서는 통과한 미션만 완료 카운트 증가
```

### 4-5. 재도전(복습) 처리

```
이미 완료한 세션 재도전 요청:
  └─▶ 복습 모드로 진입
  └─▶ 채점 후 xp = Math.floor(정상XP * 0.5)  // XP 절반
  └─▶ streak / ticket 보상 없음
  └─▶ Response에 "mode": "review" 포함
```

### 오프라인 대응 (FE)

```
네트워크 없음 감지 시:
  └─▶ 마지막 캐싱된 문제 사용 (Room DB, 최대 10문제)
  └─▶ 캐시 없으면 "인터넷 연결이 필요해요" 화면 표시
  └─▶ 오프라인 완료 결과는 큐에 저장 → 온라인 복구 시 자동 전송
```

## 5. FEATURE 02 — AI 펫 성장 게임 🐣

### 담당: FE (임다빈) — Lottie 애니메이션·UI | BE (문한일) — XP·레벨 계산

### 5-1. XP 구조 ⚠️ (BE + FE 모두 반드시 동일하게 구현)

> XP는 두 종류로 완전히 분리됩니다.
> 

```jsx
[1] pet.xp — 장착한 펫 개별 XP (리셋 O)
  알(Egg):       0   ≤ pet.xp < 80
  성장(Growth):  80  ≤ pet.xp < 250
  아이몽(Final): pet.xp ≥ 250
    └─▶ 아이몽 달성 시 → pet.xp 리셋(0) + 왕관 영구 해금 + 등급별 티켓 보상 지급
  ⚠️ 장착한 펫만 XP 획득. 비장착 펫은 XP 변동 없음.
  ⚠️ 장착 변경 시 기존 펫 XP 유지 (초기화 없음)
  ⚠️ 펫 장착 UI → 도감 화면에서 처리 (Post-MVP). MVP에서는 스타터 펫 자동 장착

[2] child.totalXp — 계정 누적 XP (리셋 없음, 3가지 역할)
  역할 1: 홈 화면 표시 — "총 XP: 320" 성취 기록
  역할 2: 데일리/위클리 퀘스트 달성 조건
  역할 3: 업적(뱃지) 달성 조건

[가챠 확률 기준 — totalXp 아님, 뽑기 횟수 기준]
  child.gachaPullCount (섹션 9 참고)
```

### 5-2. 성장 단계 명세

## 9. FEATURE 06 — 가챠 뽑기 시스템 🎰

### 담당: BE (문한일) — 확률 계산·DB | FE (임다빈) — 연출 애니메이션

⚠️ **난수 생성은 반드시 서버 사이드에서 처리** (클라이언트 조작 방지)

### 9-1. 티켓 종류 & 획득 경로

| 티켓 종류 | 보장 등급 | 획득 방법 |
| --- | --- | --- |
| 일반 티켓 | 없음 (확률 그대로) | 데일리 3개 완료, 레벨업 |
| 레어 티켓 | 레어 이상 보장 | 7일 스트릭 달성 |
| 에픽 티켓 | 영웅 이상 보장 | 30일 스트릭 달성 |

```jsx
[티켓 지급 시점 — 섹션 11 연쇄 이벤트에서 통합 처리]
// ⚠️ 데일리 퀘스트 3개 완료 티켓은 데일리 퀘스트 시스템(섹션 5-1-1)으로 통합
//    기존 "미션 3개 완료 → 티켓" 로직과 중복이므로 섹션 11 ⑧에서 제거
7일 스트릭:      tickets.rare += 1
30일 스트릭:     tickets.epic += 1
레벨업:          tickets.normal += 2
복귀 보상(3일+): tickets.normal += min(daysMissed - 2, 3)
  → 앱 오픈 시 자동 지급 X, 팝업으로 표시 후 [받기] 탭 시 지급
  → 팝업 문구: "N일 만에 돌아왔어요! 🎉 티켓 N장 드릴게요!"
스타터 패키지:   최초 앱 설치 시 tickets.normal += 3 (온보딩 완료 시점에 1회 지급)
```

### 9-2. 가챠 실행 API

```jsx
POST /api/gacha/pull
  Body: { childId, ticketType: "NORMAL" | "RARE" | "EPIC" }

[서버 처리 순서]
1. 티켓 보유 확인 → 없으면 400 에러
2. 트랜잭션 시작
3. 티켓 1장 차감 (tickets 테이블 update)
4. child.gachaPullCount += 1  // 누적 뽑기 횟수 증가
5. gachaPullCount로 확률 구간 계산
   - 0~19:  Lv.1 / 20~49: Lv.2 / 50~99: Lv.3 / 100+: Lv.4
6. 티켓 종류에 따라 확률 보정 — 제외 후 나머지 등급 비율대로 재정규화
   // NORMAL 티켓: 기본 확률표 그대로 사용

   // RARE 티켓: 일반 0% 보장, 고정값 적용 (뽑기 횟수 무관)
   //   희귀 72%, 영웅 23%, 전설 5%

   // EPIC 티켓: 일반+희귀 0% 보장, 고정값 적용 (뽑기 횟수 무관)
   //   영웅 93%, 전설 7%

   // 확률 상승 규칙 (섹션 9-4 참고)
   srBonus = Math.floor(srMissCount / 10) * 10  // 10번 단위 +10%씩
   영웅+ 확률 = min(기본영웅+ + srBonus, 100)   // 100% 초과 방지
   일반 확률  = max(0, 기본일반 - srBonus)       // 음수 방지
   // 일반이 0%가 된 경우: RARE 티켓과 동일하게 나머지 등급 재정규화

7. 확률 보정 적용 후 가중 랜덤으로 등급 결정
8. 가중 랜덤으로 등급 결정 → 해당 등급 내에서 균등 랜덤으로 캐릭터 결정
9. 구간 레벨업 체크 (이전 횟수 구간 != 현재 구간)
   → 레벨업이면: tickets.normal += 2, shieldCount += 1
10. 중복 여부 확인 → 중복이면 조각으로 변환
    → 신규이면: pets 테이블에 {petId, childId, xp: 0, stage: EGG} 추가
11. 결과 DB 저장
12. 트랜잭션 커밋
13. 결과 반환

Response:
{
  "result": {
    "petId": "pet_rare_003",
    "petName": "번개몽",
    "grade": "RARE",
    "isNew": false,
    "fragments": 3         // 중복이면 조각 수, 신규면 0
  },
  "srMissCount": 8,         // 현재 영웅 이상 미획득 연속 횟수 (10 이상이면 FE에 확률업 표시)
  "srBonus": 0,             // 현재 적용 중인 영웅+ 확률 보너스 (srMissCount/10*10)
  "remainingTickets": 2
}
```

### 9-3. 뽑기 횟수 연동 동적 확률표

> 기준: `child.gachaPullCount` (누적 총 뽑기 횟수, 리셋 없음, totalXp 무관)
>

### 9-4. 확률 상승 시스템 (구 천장 방식 대체)

```jsx
// ⚠️ 기존 소프트/하드 천장 완전 제거
// → 10번 단위 누적 상승 + 영웅 이상 뽑히면 즉시 리셋으로 대체

[확률 상승 규칙]
  srBonus = Math.floor(srMissCount / 10) * 10
  // srMissCount 0~9:   srBonus = 0%   (기본 확률 그대로)
  // srMissCount 10~19: srBonus = 10%
  // srMissCount 20~29: srBonus = 20%
  // srMissCount 30~39: srBonus = 30%  ... 무한 누적

  영웅+ 확률 = min(기본영웅+ + srBonus, 100)  // 100% 초과 방지
  일반 확률  = max(0, 기본일반 - srBonus)      // 음수 방지
  // 일반이 0%가 된 경우 → RARE 티켓과 동일하게 나머지 등급 재정규화

[리셋 조건]
  영웅 이상 획득 시 → srMissCount = 0 즉시 리셋
  // 하드 천장(50번 보장) 없음 — srBonus 무한 누적으로 대체

[예시]
  뽑기 15번째: srBonus = 10% → Lv.1 기준 영웅+ 확률 10% → 19%
  뽑기 25번째: srBonus = 20% → Lv.1 기준 영웅+ 확률 20% → 29%
  영웅 뽑힘!  → srMissCount = 0, 다음 뽑기부터 기본 확률

[FE 표시]
  srMissCount >= 10 일 때만 표시:
  "💜 영웅 확률 +{srBonus}% 업 중! ({srMissCount}번째 도전 중)"
  srMissCount < 10: 표시 없음
```

  OFFLINE           // 오프라인 (캐시 문제 사용 중 배너 표시)
}
